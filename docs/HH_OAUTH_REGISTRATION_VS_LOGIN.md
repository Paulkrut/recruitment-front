# OAuth через HH - Регистрация vs Авторизация

## 🎯 Главное: Одна кнопка = Два действия

**OAuth через HeadHunter работает И для регистрации И для авторизации одновременно!**

```
User → "Войти через HH"
  ↓
Backend проверяет: есть ли User с таким email?
  ↓
  ├─ ✅ Есть → АВТОРИЗАЦИЯ (merge accounts если нужно)
  └─ ❌ Нет → РЕГИСТРАЦИЯ (создать User + Company + Employee)
```

---

## 📊 Сравнение

### **Обычная регистрация (по email):**
```
1. User заполняет форму (имя, email, компания, должность)
2. Backend:
   - Создаёт User
   - Создаёт Company
   - Создаёт Employee
   - Отправляет пароль на email
3. User получает email с паролем
4. User вводит email + пароль для входа
```

### **Регистрация через HH OAuth:**
```
1. User нажимает "Войти через HH"
2. HH авторизует пользователя
3. Backend:
   - Получает профиль из HH (email, имя, компания из employer)
   - Создаёт User (автоматически!)
   - Создаёт Company (название из HH employer)
   - Создаёт Employee (имя из HH профиля)
   - Генерирует JWT токен
4. User сразу вошёл в систему ✅
```

**Преимущества OAuth регистрации:**
- ✅ Не нужно заполнять форму
- ✅ Не нужно ждать email с паролем
- ✅ Автоматическое заполнение данных из HH
- ✅ Мгновенный вход после регистрации

## 🔍 Логика Backend

### **HhOAuthService::processCallback()**

```php
public function processCallback(array $tokenData): array
{
    // 1. Получаем HH профиль
    $hhProfile = $this->getUserProfile($tokenData['access_token']);
    $email = $hhProfile['email'];
    $hhUserId = $hhProfile['id'];

    // 2. Проверяем: есть ли User с таким email?
    $existingUser = $this->userRepository->findByEmail($email);

    if ($existingUser) {
        // АВТОРИЗАЦИЯ - пользователь уже есть
        
        // Проверяем: email подтверждён?
        if (!$existingUser->isEmailVerified()) {
            return ['error' => 'email_not_verified', 'email' => $email];
        }

        // Связываем HH аккаунт с существующим User
        $this->linkHhAccountToUser($existingUser, $hhUserId, $hhProfile, $tokenData);

        return [
            'user' => $existingUser,
            'is_new' => false,
        ];
    }

    // 3. РЕГИСТРАЦИЯ - создаём нового пользователя
    $user = $this->createUserFromHhProfile($hhProfile);
    $this->linkHhAccountToUser($user, $hhUserId, $hhProfile, $tokenData);

    return [
        'user' => $user,
        'is_new' => true,
    ];
}
```

## 📝 Где используется кнопка

### **1. Страница авторизации (`/auth/login`)**
```tsx
<HhLoginButton 
  variant="outlined"
  fullWidth
/>
```

**Текст:** "Войти через HeadHunter"

**Поведение:**
- Если User есть → авторизует
- Если User нет → регистрирует + авторизует

---

### **2. Страница регистрации (`/auth/register`)**
```tsx
<HhLoginButton 
  variant="outlined"
  fullWidth
/>
<Typography variant="caption" color="text.secondary">
  Быстрая регистрация через HeadHunter - данные заполнятся автоматически
</Typography>
```

**Текст:** "Войти через HeadHunter"

**Поведение:**
- Если User есть → авторизует (просто войдёт)
- Если User нет → регистрирует + авторизует ✅

**Почему тот же текст?**
- Backend всё равно проверит: есть User или нет
- OAuth концептуально = "войти" (неважно есть аккаунт или нет)
- HH также использует "Войти" а не "Зарегистрироваться"

## 🎨 UX объяснение

### **На странице регистрации:**

```
┌─────────────────────────────────────────┐
│          Регистрация                    │
├─────────────────────────────────────────┤
│                                         │
│  [Войти через HeadHunter]               │ ← Быстрая регистрация
│  Быстрая регистрация через HeadHunter   │
│  - данные заполнятся автоматически      │
│                                         │
│  ─────── или заполните форму ───────    │ ← Разделитель
│                                         │
│  [Имя]           ___________________    │
│  [Должность]     ___________________    │
│  [Компания]      ___________________    │
│  [Email]         ___________________    │
│  [Телефон]       ___________________    │
│                                         │
│  [Зарегистрироваться]                   │ ← Обычная регистрация
│                                         │
└─────────────────────────────────────────┘
```

**User flow:**
1. User видит кнопку HH вверху
2. Понимает: "О, можно быстро через HH!"
3. Нажимает → авторизуется на HH → мгновенно зарегистрирован и вошёл ✅

**Альтернативный flow:**
1. User не хочет через HH
2. Заполняет форму вручную
3. Регистрируется → получает email → вводит пароль

## ✅ Преимущества OAuth регистрации

### **Для пользователя:**
- ⚡ Мгновенно (1 клик вместо заполнения формы)
- 🔐 Безопасно (не нужно придумывать пароль)
- 📧 Не нужно ждать email
- ✅ Данные заполняются автоматически из HH

### **Для нас:**
- ✅ Меньше брошенных регистраций
- ✅ Проверенные email (HH проверил)
- ✅ Данные из HH (название компании, employer)
- ✅ Связь с HH (потенциал для интеграции вакансий)

## 🔄 Сценарии

### **Сценарий 1: Новый пользователь регистрируется через HH**
```
1. User на /auth/register
2. Нажимает "Войти через HH"
3. Авторизуется на HH
4. Backend:
   - Email: ivan@company.ru (из HH)
   - НЕТ такого User в базе
   - Создаём User, Company, Employee
   - Генерируем JWT
5. User сразу в системе ✅
```

### **Сценарий 2: Существующий пользователь (зарегистрирован по email) входит через HH**
```
1. User на /auth/login
2. Нажимает "Войти через HH"
3. Авторизуется на HH
4. Backend:
   - Email: ivan@company.ru (из HH)
   - ✅ ЕСТЬ такой User в базе
   - Email подтверждён? ✅ Да
   - Связываем HH аккаунт с User
   - Генерируем JWT
5. User вошёл в систему ✅
```

### **Сценарий 3: User пытается войти через HH, но email не подтверждён**
```
1. User на /auth/login
2. Нажимает "Войти через HH"
3. Авторизуется на HH
4. Backend:
   - Email: ivan@company.ru (из HH)
   - ✅ ЕСТЬ такой User в базе
   - Email подтверждён? ❌ НЕТ
   - Возвращаем ошибку: email_not_verified
5. Frontend показывает:
   ⚠️ "Ваш email ivan@company.ru ещё не подтверждён.
   Для объединения с аккаунтом HH сначала подтвердите email."
```

## 📋 Итог

**Одна кнопка "Войти через HH" используется:**
- ✅ На странице авторизации (`/auth/login`)
- ✅ На странице регистрации (`/auth/register`)

**Backend сам определяет:**
- Если User с таким email уже есть → **авторизация**
- Если User нет → **регистрация**
